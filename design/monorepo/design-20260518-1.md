# Portfolio Monorepo Shell Design

## Summary

The Portfolio project should become the home surface for Timeline, Chess, and future app experiences. The recommended structure is a conventional frontend monorepo with one deployable Next.js shell app and multiple reusable internal packages.

The existing worktree layout should remain intact after the local parent directory rename: `portfolio/dev` is the active worktree, and `portfolio/.main/` is the hidden main worktree. The remote repository rename is a separate repository-hosting concern and can be handled independently.

Target URL behavior:

```text
/                         Portfolio home page
/apps/timeline            Timeline app experience
/apps/timeline/chess      Chess embedded in Timeline layout context
/apps/chess               Chess standalone/fullscreen experience
```

## Goals

- Preserve the existing Git repository and useful Timeline frontend history.
- Use Portfolio as the single user-facing Next.js shell app.
- Keep technical package boundaries clear through workspaces.
- Support Timeline-context and standalone-context Chess routes without duplicating Chess implementation.
- Allow future apps to be added under `/apps/*` without proxy, iframe, or subpath-mount complexity.
- Keep `portfolio/dev` and `portfolio/.main/` as worktree directories.

## Current State

The frontend now lives in the Portfolio local worktree layout:

```text
portfolio/
  dev/       active frontend worktree
  .main/     hidden main worktree
```

Timeline remains one app experience inside the Next.js frontend implementation, including feature folders such as Chess. The desired product direction changes the frontend from a Timeline-only app into a Portfolio shell that can host Timeline, Chess, and future app experiences.

The backend remains separate:

```text
timeline-backend/dev
```

This separation is acceptable because the backend has distinct runtime, deployment, and service concerns.

## Proposed Design

### Repository Model

Continue using the existing frontend Git repository. Rename the repository identity to Portfolio when appropriate, but avoid disrupting local worktree semantics.

Conceptually:

```text
timeline repository  ->  portfolio repository
```

The local workspace has been renamed to match the Portfolio identity:

```text
/root/Desktop/portfolio/dev
```

The remote repository rename remains a separate hosting operation.

### Monorepo Shape

Use the conventional monorepo layout:

```text
dev/
  apps/
    portfolio/
      app/
      public/
      next.config.ts
      package.json
  packages/
    chess/
      package.json
      src/
    timeline/
      package.json
      src/
    ui/
      package.json
      src/
    auth/
      package.json
      src/
    api-client/
      package.json
      src/
    config/
      package.json
  package.json
  bun.lock
  turbo.json
```

`apps/portfolio` is the only deployable frontend app. It owns the Next.js route tree, global app providers, metadata, top-level layouts, and deployment configuration.

`packages/*` are internal workspace packages. They provide app features, shared UI, backend clients, and configuration.

### Route Ownership

The Portfolio shell owns the URL tree:

```text
apps/portfolio/app/page.tsx
apps/portfolio/app/apps/timeline/page.tsx
apps/portfolio/app/apps/timeline/chess/page.tsx
apps/portfolio/app/apps/chess/page.tsx
```

Expected route mapping:

```text
apps/portfolio/app/page.tsx                         -> /
apps/portfolio/app/apps/timeline/page.tsx           -> /apps/timeline
apps/portfolio/app/apps/timeline/chess/page.tsx     -> /apps/timeline/chess
apps/portfolio/app/apps/chess/page.tsx              -> /apps/chess
```

The folder name `apps/portfolio` is a monorepo implementation detail and does not add `/portfolio` to the URL.

### Package Responsibilities

#### `packages/chess`

Owns the reusable Chess feature:

- Chess shell component.
- Chess board UI.
- Chess controls.
- Chess hooks and store.
- Chess API client calls.
- Chess-specific types and helpers.

The package should expose a context-aware entry point:

```tsx
<ChessGameShell variant="standalone" />
<ChessGameShell variant="timeline-embedded" />
```

The `standalone` variant renders fullscreen. The `timeline-embedded` variant adjusts density, spacing, and chrome to fit inside Timeline layout with `TimelineNavigationBar`.

#### `packages/timeline`

Owns Timeline feature and layout primitives:

- `TimelineLayout`.
- `TimelineNavigationBar`.
- Timeline app pages/components that are not route files.
- Timeline-specific state and UI composition.

The Portfolio route decides when Timeline layout wraps another feature:

```tsx
<TimelineLayout>
  <ChessGameShell variant="timeline-embedded" />
</TimelineLayout>
```

#### `packages/ui`

Owns shared UI primitives:

- shadcn/Radix wrappers.
- `Button`, `Dialog`, `Select`, `Tooltip`, and other reusable components.
- `cn()` and styling helpers.
- shared theme tokens where practical.

#### `packages/api-client`

Owns typed backend calls that are not tied to a route context:

- Timeline backend fetch helpers.
- Chess API helpers.
- Shared response types and request builders.

When auth, cookies, or headers are needed, route-level Next.js code can pass explicit configuration into these clients.

#### `packages/auth`

Owns shared auth primitives and client-side auth state. If Next.js-specific server helpers are needed, expose them deliberately through a clear path such as:

```ts
@portfolio/auth/next
```

This keeps the package structure honest without forbidding Next.js features.

#### `packages/config`

Owns shared tooling configuration:

- TypeScript base configs.
- ESLint shared config.
- Tailwind preset or shared theme config if useful.

### Next.js Feature Usage in Packages

Because Portfolio intentionally uses one Next.js shell app, internal packages do not need to pretend they are framework-agnostic. Packages can use Next.js features when that is the cleanest design.

The constraint is architectural clarity, not framework avoidance:

- Pure logic should stay pure when possible.
- React UI packages can be React-specific.
- Next.js-specific helpers should be explicit and not accidentally imported by pure modules.

A practical package can contain both generic and Next-specific areas:

```text
packages/chess/src/core/       pure helpers
packages/chess/src/components/ React components
packages/chess/src/next/       Next-specific helpers, only if needed
```

This allows the project to leverage Next.js fully while keeping module boundaries understandable.

### Dependency Ownership

Use workspaces. The root owns orchestration and shared tooling. Each package owns its direct dependencies.

Root package responsibilities:

- workspace definitions
- Turbo scripts
- shared developer commands
- root lockfile

`apps/portfolio` responsibilities:

- `next`
- `react`
- `react-dom`
- route-level app dependencies

Feature package responsibilities:

- `packages/chess` owns chess-specific dependencies such as `chess.js`.
- `packages/ui` owns UI primitive dependencies.
- `packages/api-client` owns client/schema dependencies.

Do not install every runtime dependency at the root by default. Dependencies should live where they are directly imported.

### Workspace Tooling

Use Bun workspaces to match the current project ecosystem.

Root `package.json` shape:

```json
{
  "private": true,
  "workspaces": [
    "apps/*",
    "packages/*"
  ],
  "scripts": {
    "dev": "turbo dev",
    "dev:portfolio": "turbo dev --filter=@portfolio/portfolio",
    "build": "turbo build",
    "typecheck": "turbo typecheck",
    "lint": "turbo lint"
  }
}
```

Internal package dependency example:

```json
{
  "dependencies": {
    "@portfolio/chess": "workspace:*",
    "@portfolio/timeline": "workspace:*",
    "@portfolio/ui": "workspace:*"
  }
}
```

`apps/portfolio/next.config.ts` should transpile workspace packages when they ship TypeScript/TSX source:

```ts
const nextConfig = {
  transpilePackages: [
    '@portfolio/chess',
    '@portfolio/timeline',
    '@portfolio/ui',
    '@portfolio/auth',
    '@portfolio/api-client',
  ],
};

export default nextConfig;
```

### Git and Worktree Strategy

The existing Git repository can continue as the canonical repository. The project can be renamed at the remote/repository metadata layer without discarding history.

Recommended constraints:

- Keep `portfolio/dev` as the active worktree directory.
- Keep `portfolio/.main/` as the hidden main worktree.
- Prefer `git mv` when moving files into `apps/portfolio` or `packages/*` so history remains easier to follow.
- Avoid copying code into a separate fresh repository unless there is a strong reason to split history.

A later remote rename can be handled with:

```bash
git remote set-url origin <new-portfolio-remote-url>
```

The local parent directory has already been renamed; keep the worktree metadata synchronized if it changes again.

## Migration Plan

1. Keep the current `portfolio/dev` active worktree.
2. Add root workspace configuration in `portfolio/dev`.
3. Move the existing Next.js app into `apps/portfolio` with `git mv` where possible.
4. Confirm the Portfolio app still runs as the only Next.js shell.
5. Add `/`, `/apps/timeline`, `/apps/timeline/chess`, and `/apps/chess` routes.
6. Extract shared UI into `packages/ui`.
7. Extract Chess into `packages/chess` and introduce `variant="standalone" | "timeline-embedded"`.
8. Extract Timeline shell/navigation into `packages/timeline`.
9. Add workspace package imports and configure `transpilePackages`.
10. Update deployment/build scripts to target `apps/portfolio`.
11. Rename repository metadata/remotes to Portfolio when ready.

## Open Questions

- Should the workspace package namespace be `@portfolio/*`, `@ashe/*`, or another organization-specific scope?
- Should `packages/auth` be split into `packages/auth` and `packages/auth-next`, or should it expose a `@portfolio/auth/next` subpath?
- Should `packages/chess` keep API calls internally, or should all backend calls move to `packages/api-client` immediately?
- Should Timeline backend API environment variables stay app-level only, or should a typed runtime config package be introduced?
- When should the remote repository be renamed from Timeline to Portfolio?
